# Wallet Signature Verification Flow

To protect critical actions (such as transferring ownership, deleting snippets, regenerating invite codes, or releasing escrow), the Codely platform requires cryptographic proof of intent via a wallet signature.

## Why is this needed?

Even with a valid authentication token, requiring a signature at the exact moment of a critical action provides:
1. **Proof of Intent:** The user explicitly approves the action.
2. **Replay Protection:** Nonce and timestamp validation prevent an attacker from reusing an old signature to repeat a destructive action.
3. **Auditing:** We can store the signature and parameters to prove that an authorized wallet signed the request.

## Implementation Details

### Signed Message Format

The frontend prompts the wallet (Freighter, Albedo, etc.) to sign a canonical string formatted exactly as follows:

```
Codely signature request
Action: {action}
Resource: {resourceId}
Nonce: {nonce}
Timestamp: {timestamp}
```

- **`{action}`**: A specific identifier for the action being performed (e.g., `delete_snippet`, `transfer_ownership`).
- **`{resourceId}`**: The ID of the affected resource (e.g., snippet ID).
- **`{nonce}`**: A unique random string generated by the client (e.g., UUID v4).
- **`{timestamp}`**: The UNIX timestamp (in milliseconds) at the time of signing.

### Backend Verification

The backend uses `SignatureMiddleware` to enforce the signature check. It expects the following headers on the incoming HTTP request:

- `x-wallet-signature`: The base64-encoded signature returned by the wallet.
- `x-wallet-nonce`: The same nonce used in the signed message.
- `x-wallet-timestamp`: The timestamp (ms) used in the signed message.
- `x-wallet-address`: The public key of the signing wallet.

#### Process:
1. Validates that the timestamp is no older than 5 minutes.
2. Reconstructs the exact message string.
3. Uses the `@stellar/stellar-sdk` to verify that the signature matches the reconstructed message and public key.
4. Returns an error (`401 Unauthorized` or `403 Forbidden`) if validation fails.

## Developer Guide

### Adding Signature Verification to an Endpoint

To require a signature for a new API route, simply call the middleware before performing the sensitive logic:

```typescript
import { SignatureMiddleware } from "../signature.middleware";

const signatureMiddleware = new SignatureMiddleware();

export async function POST(req: NextRequest) {
  // Extract inputs
  const resourceId = "123";
  const actionName = "transfer_ownership";

  // Verify wallet signature
  const signatureResult = await signatureMiddleware.verifySignature(req, actionName, resourceId);
  if (!signatureResult.isValid) {
    return signatureResult.error!;
  }
  
  // Proceed with sensitive action...
}
```

### Triggering a Signature from the Frontend

The `WalletContext` provides a helper method `signAction` that handles the nonce/timestamp generation and wallet prompting:

```tsx
import { useWallet } from "@/components/WalletConnect";

export function DeleteButton({ snippetId }) {
  const wallet = useWallet();

  const handleDelete = async () => {
    try {
      // 1. Prompt user for signature
      const signatureData = await wallet.signAction("delete_snippet", snippetId);
      
      // 2. Send request with headers
      await fetch(`/api/snippets/${snippetId}`, {
        method: "DELETE",
        headers: {
          "x-wallet-signature": signatureData.signature,
          "x-wallet-nonce": signatureData.nonce,
          "x-wallet-timestamp": signatureData.timestamp.toString(),
          "x-wallet-address": wallet.publicKey
        }
      });
    } catch (error) {
      console.error("Action failed or signature rejected", error);
    }
  }

  return <button onClick={handleDelete}>Delete Snippet</button>;
}
```
